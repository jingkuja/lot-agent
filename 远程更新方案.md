# Lot Agent 远程更新方案（方案 A · 整镜像 tar 包 · 盒子主动拉取）

> 文档日期：2026-06-25
> 适用范围：**仅应用（app）远程更新** —— `server` / `worker` / `web` 三个容器的镜像化更新。
> postgres / redis 当前随盒子部署，**后续上云**；本方案不更新数据容器，但架构已为其上云预留。

---

## 0. 部署约束（决定整个方案）

| 约束 | 含义 |
|---|---|
| 计算盒子在**客户现场** | 你方运维无法物理接触 |
| **无法主动连入盒子**（无入站、无 SSH） | ❌ 一切"推送"式更新（`docker save \| ssh`、远程触发）作废 |
| 盒子**可出网**访问你方服务器：调用更新 API、下载文件 | ✅ 只能走**盒子主动拉取（phone-home）**模型 |
| 进不去盒子排错 | ✅ 更新必须**自愈**：失败自动回滚 + 回报状态 |

由这些约束，以下设计点是**定死的，不再可选**：

1. **触发只能是盒子轮询**：盒子上有 `updater` 进程，定时回连更新 API 查版本。
2. **必须自动回滚**：更新后健康检查失败 → 自动切回上一个版本，绝不让盒子卡在坏版本。
3. **只更新 app 容器**：`server` / `worker` / `web`；不碰 `postgres` / `redis`（上云后盒子只剩 app + updater，模型不变）。
4. **产物 = 整镜像 tar 包**（方案 A）：`docker save` 出的 `lot-agent-app` / `lot-agent-web` 镜像包，盒子 `docker load` 后切 tag 重启。代码+依赖永远一致、回滚最干净、产物单一。

---

## 1. 总体架构

```
┌──────────────────── 你方中心服务器（更新服务端） ─────────────────────┐
│  GET  /update/manifest      → 返回最新版本清单（JSON）                  │
│  GET  /artifacts/*.tar.gz   → 镜像 tar 包静态下载（带 sha256）          │
│  POST /update/report        → 接收盒子上报（当前版本/更新结果/健康）    │
│  （最小实现：HTTPS + 静态文件 + 一个清单端点 + 一个上报端点）          │
└──────────────────────────────────▲───────────────────────────────────┘
                  出网 HTTPS（盒子 → 服务器，单向）│
┌──────────────────────────────────┴─── 客户盒子 ───────────────────────┐
│  updater（新增容器）：轮询清单 → 下载校验 → docker load → 切 tag       │
│                       → compose up app → 健康检查 → 成败/回滚 → 上报    │
│        挂载 /var/run/docker.sock + compose 目录 + 状态卷               │
│                                                                       │
│  app（被更新）：  server      worker      web                          │
│  data（不更新）： postgres    redis        ← 后续上云后从盒子移除      │
└───────────────────────────────────────────────────────────────────────┘
```

**更新服务端是你方的中心服务器，与盒子里的 `server` 容器是两回事**，不要混淆。它对盒子而言就是"更新 API + 文件下载"，因此**不需要 docker registry**。

## 2. 组件职责

### 2.1 更新服务端（你方中心服务器）

| 端点 | 方法 | 作用 |
|---|---|---|
| `/update/manifest` | GET | 返回"当前应发布的版本清单"。盒子带 `box_id` / `channel` / 鉴权 token。 |
| `/artifacts/<file>.tar.gz` | GET | 镜像 tar 包静态下载。建议支持 `Range`（断点续传，弱网友好）。 |
| `/artifacts/docker-compose-<ver>.yml` | GET | （可选）当 compose 结构变化时下发新编排。 |
| `/update/report` | POST | 盒子上报：当前版本、上次更新结果、健康状态、时间戳。 |

最小实现：一个 nginx/对象存储托管 `artifacts/` + `manifest.json`，外加一个轻量服务处理 `report`。**鉴权**：盒子在请求头带 `X-Box-Token`，服务端校验，确保只有你的盒子能取制品。

### 2.2 盒子上的 updater（本方案新增的唯一组件）

一个**极简、极稳定**的小容器：
- 挂载 `/var/run/docker.sock` → 能 `docker load` / `docker compose up`（即 Watchtower 类一体机更新代理的标准做法）。
- 挂载盒子上的 compose 目录（含 `docker-compose.yml`、`.env`）。
- 挂载一个 `updater_state` 卷，持久化 `current_version` / `previous_version` / 上次结果。
- 内含 `box_id` 与 `X-Box-Token`（来自 `.env`，不写死在镜像）。

> **updater 自身要尽量稳定、少改动**。它的自更新是鸡生蛋问题，列为后续考虑（§9），首版假定 updater 长期不变。

## 3. 版本清单（manifest）格式

```json
{
  "version": "1.2.0",
  "released_at": "2026-06-25T10:00:00Z",
  "mandatory": false,
  "images": [
    {
      "name": "lot-agent-app",
      "tag": "1.2.0",
      "url": "https://update.example.com/artifacts/lot-agent-app-1.2.0.tar.gz",
      "sha256": "e3b0c44298fc1c149afbf4c8996fb924...",
      "size": 312000000
    },
    {
      "name": "lot-agent-web",
      "tag": "1.2.0",
      "url": "https://update.example.com/artifacts/lot-agent-web-1.2.0.tar.gz",
      "sha256": "a1d0c6e83f027327d8461063f4ac58a6...",
      "size": 18000000
    }
  ],
  "compose_url": null,
  "notes": "修复标题生成；升级 server 健康检查"
}
```

- **一个 release 版本号统一管两个镜像**（app + web 同版本号发布），简单、避免错配。
- `compose_url` 平时为 `null`（只换镜像 tag）；**仅当 compose 结构变化**（新增服务/环境变量）时下发新编排文件。
- `mandatory`：保留位，用于将来"强制升级"策略。
- `sha256` / `size`：完整性校验与进度/磁盘预检。

## 4. 更新流程（updater 主循环）

```
循环（每 UPDATE_INTERVAL，建议 5~15 分钟一次）：

1. GET /update/manifest（带 box_id + token）
2. 读本地状态 current_version
   manifest.version == current_version ?  → 是：结束本轮
3. 否（有新版本）：
   a. 磁盘预检（size 之和 + 余量）；不足 → 放弃 + 上报
   b. 逐个下载镜像 tar 包到临时目录 → 校验 sha256
        任一失败 → 放弃，保持旧版本运行 → 上报 failure
   c. docker load 每个 tar 包（镜像带新 tag，旧 tag 镜像保留不删）
   d. previous_version = current_version  （供回滚）
   e. 切换编排到新 tag：
        - 常规：改 .env 的 APP_TAG=1.2.0（compose 用 ${APP_TAG}）
        - compose 结构变化：先拉 compose_url 覆盖 docker-compose.yml
   f. docker compose up -d server worker web   （只重建 app；data 容器不动）
   g. 健康检查（最多 HEALTH_TIMEOUT，例如 90s）：
        - web   :  GET http://web/healthz        == 200
        - server:  容器内 GET :3000/health        == ok
        全部通过 ?
          ├─ 是：current_version = 1.2.0；清理过旧镜像（保留最近 1~2 版）→ 上报 success
          └─ 否：**回滚** ↓
4. 回滚：
   a. 切回 previous_version 的 tag（恢复 .env / 必要时恢复旧 compose）
   b. docker compose up -d server worker web
   c. 复测健康；无论结果都上报 rolled_back + 失败详情（日志摘要）
```

关键不变量：
- **data 容器全程不动**：`docker compose up -d server worker web` 显式只列 app 服务。
- **旧镜像不立即删**：回滚要靠它，至少保留上一版（§6）。
- **任何一步失败都不让盒子停在坏状态**：要么没更成（旧版继续跑），要么回滚回旧版。

---

## 5. 健康检查与回滚（进不去盒子，这是命门）

- 健康判定 = **web 反代存活** + **server `/health` 通**（server 起得来即意味着数据库连接 OK、迁移跑完）。可选加一条"经 nginx 访问受保护 API 返回 401"证明全链路通。
- 回滚之所以可靠，是因为方案 A 是**不可变镜像**：上一版镜像还在盒子本地，回滚只是 `docker compose up -d` 切回旧 tag，**秒级、确定性**。
- **健康检查必须有超时**，超时即判失败回滚，避免盒子卡在"启动中"。
- **迁移与回滚的约束（重要）**：代码回滚**不会回滚数据库迁移**。因此迁移必须**只增不改、向后兼容**（沿用现有 `IF NOT EXISTS` 幂等约定），保证"旧代码也能跑在新库结构上"。否则回滚后旧代码遇到新结构会出错——这是无人值守盒子上最隐蔽的坑。

## 6. 盒子本地状态与磁盘

`updater_state` 卷中维护一个 `state.json`：

```json
{
  "current_version": "1.2.0",
  "previous_version": "1.1.0",
  "last_check_at": "2026-06-25T10:05:00Z",
  "last_result": "success",
  "last_error": null
}
```

- **镜像保留策略**：本地保留最近 **2** 个版本的镜像（当前 + 上一版），更老的在更新成功后 `docker image prune` 清理，控制盒子磁盘。
- 状态卷随容器销毁不丢，盒子重启后 updater 据此继续。

## 7. 编排改动（盒子上的 docker-compose.yml）

在现有 5 服务基础上，**镜像 tag 参数化** + **新增 updater**：

```yaml
# app 镜像 tag 由 .env 注入，updater 改 .env 即完成切版
services:
  server:
    image: lot-agent-app:${APP_TAG:-1.0}
  worker:
    image: lot-agent-app:${APP_TAG:-1.0}
  web:
    image: lot-agent-web:${WEB_TAG:-1.0}

  updater:
    image: lot-agent-updater:1.0     # 极简稳定，长期不变
    restart: unless-stopped
    environment:
      UPDATE_SERVER: ${UPDATE_SERVER}        # https://update.example.com
      BOX_ID: ${BOX_ID}
      BOX_TOKEN: ${BOX_TOKEN}
      UPDATE_INTERVAL: ${UPDATE_INTERVAL:-600}
      HEALTH_TIMEOUT: ${HEALTH_TIMEOUT:-90}
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock   # 控制 app 容器
      - ./:/workspace                                # compose 目录 + .env
      - updater_state:/state
```

> `server` / `worker` / `web` 的现有 Dockerfile、entrypoint、nginx 反代**完全沿用**，本方案不改它们；只是 tag 改为变量驱动，并加入 updater 与编排逻辑。

## 8. 你方发布流程（构建侧 `release.sh`）

在你的构建机（你本地或 CI）上：

```bash
VER=1.2.0
# 1) 构建（沿用现有两份 Dockerfile）
docker build -t lot-agent-app:$VER -f Dockerfile .
docker build -t lot-agent-web:$VER -f Dockerfile.web .
# 2) 导出为 tar 包
docker save lot-agent-app:$VER | gzip > lot-agent-app-$VER.tar.gz
docker save lot-agent-web:$VER | gzip > lot-agent-web-$VER.tar.gz
# 3) 算 sha256 / size，生成 manifest.json
# 4) 上传 tar 包 + manifest 到更新服务端（rsync / 对象存储 / 内部 API）
#    —— 上传是“最后一步、原子切换”：先传 tar 包，最后再发布新 manifest，
#       避免盒子读到 manifest 却下不到文件。
```

**发布顺序很关键**：先上传所有制品，**最后**才更新 `manifest.json` 指向新版本；这样盒子永远不会拉到"清单已变、文件还没到"的半成品。

## 9. 安全

- manifest 与制品全程 **HTTPS**。
- **sha256 强制校验**：防半包/损坏，校验不过直接放弃本轮更新。
- **盒子鉴权**：请求头 `X-Box-Token`，服务端校验，仅你的盒子可取制品。
- **（建议，可二期）清单签名**：你方私钥对 manifest 签名，updater 用内置公钥验签，防制品被篡改/中间人下发恶意镜像——客户现场无人值守，这一层价值高。
- **docker.sock 风险**：updater 拥有 docker.sock 等于盒子上的高权限，必须保持 updater 代码极简、来源可信、最好走签名校验。

## 10. 范围与后续演进

**本方案做的**：`server` / `worker` / `web` 三个 app 容器的镜像化远程更新（盒子拉取、自愈回滚、状态上报）。

**本方案不做（后续）**：
- **数据库上云**：postgres / redis 迁到云端托管后，从盒子 compose 移除；app 容器改连云端连接串（走 `.env`）。届时盒子只剩 app + updater，本方案模型不变。
- **updater 自更新**：首版假定 updater 不变；将来若要更新它，需设计"updater 拉取自身新镜像并自替换"机制（鸡生蛋），列为二期。
- **灰度/分批发布**：manifest 现支持 `box_id` / `channel`，可扩展为按盒子分批放量、强制升级策略。
- **清单签名验签**：见 §9，建议二期补上。
- 操作系统/宿主机层更新、updater 之外的现场运维，不在本方案。
